#+TITLE: Collaborative Ticketing System - User Experience Specification
#+DATE: 2025-11-30

* Overview

A collaborative ticketing system where users can assign tasks to each other and manage their own workload. Unlike traditional ticketing systems, this enables bidirectional task assignment - friends can ticket each other for various needs, from reading recommendations to urgent favors. The system supports multiple front-end interfaces (web, mobile, CLI) backed by a single REST API.

* Core Concepts

** Tickets
- Unique identifier and title
- Description with rich text support
- Priority levels: Low, Normal, High, Urgent
- Status: Open, In Progress, Blocked, Completed, Cancelled
- Assignee (any user in the system - you can assign tickets to friends)
- Assigner (person who created and assigned the ticket)
- Due date (optional)
- Three-tier categorization system: Category / Type / Item (e.g., home / cleaning / kitchen)
- Comments thread for updates

** Users
- Registered users (can assign and receive tickets)

** Categories (CTI System)
Three-tier categorization where users can create new CTIs:
- Category: Broad domain (Personal Development, Favors & Errands, Social, Work, Health)
- Type: Specific area within category (reading, cleaning, meetings, exercise)
- Item: Granular task type (article, kitchen, standup, cardio)

For prototype: Any user can create new CTI combinations

* Key Collaborative Features

** Bidirectional Assignment
- Any user can assign tickets to any other user on the instance
- Clear visibility into who assigned what to whom
- Ability to delegate received tickets to others (with original assigner notification)

** Single Instance User Model
- All registered users on an instance can assign tickets to each other
- No friend request system needed - implicit trust within instance
- Future consideration: Instance federation for cross-server assignments

* User Stories

** As a user, I want to:
- View tickets assigned to me in a personalized dashboard
- View tickets I've assigned to others to track their progress
- Create tickets and assign them to friends for various needs
- Set priority levels and due dates when assigning tickets
- Update status on tickets assigned to me
- Comment on tickets for updates and clarification
- Delegate tickets assigned to me to other mutual friends
- Filter and search across all tickets I'm involved with
- Create self-assigned tickets for personal task management

** As someone receiving a ticket assignment, I want to:
- Receive clear notifications about new assignments
- Understand the context, priority, and expectations
- Be able to ask clarifying questions via comments
- Update the assigner on my progress
- Mark tickets complete when finished
- Decline assignments if I'm unable to help (with explanation)


* User Interface Flows

** Dashboard (Main View)
Two-tab layout:
- "Assigned to Me": Tickets others have assigned to me
- "Assigned by Me": Tickets I've assigned to others

Each tab includes:
- Summary widgets: Open count, overdue items, recent activity
- Ticket list with columns (ID, Title, Priority, Status, Due Date, Assigner/Assignee)
- Quick actions (Update Status, Add Comment, View Details)
- Filter sidebar: Status, Priority, CTI categories, Date ranges, People
- Search across titles and descriptions

** Ticket Detail View
- Full ticket information with edit capabilities
- Comment timeline with timestamps
- Related tickets or dependencies
- File attachments support
- Status change history

** Ticket Creation/Assignment
- Select assignee from user list on the instance
- Simple form: title, description, priority (Sev-1 through Sev-2.5 through Sev-4)
- CTI categorization with autocomplete and ability to create new combinations
- Optional fields: due date, tags
- Markdown editor for descriptions
- File attachment capability
- Assignment preview before sending

** User Management
- View all users on the instance for ticket assignment
- Set personal preferences:
  - Maximum concurrent assignments: Number limit (optional)
  - Notification preferences for different users
  - "Do not disturb" hours for non-urgent tickets

* Priority System

** Sev-1
- Immediate attention required (emergencies, time-sensitive favors)
- Email/SMS notifications
- Appears at top of all views

** Sev-2
- Important but not emergency
- Should be addressed within 24-48 hours
- Push notifications
** Sev-2.5
- About as important as Sev-2, but should not alert the assignee during their night-time hours
- Users can set their "daytime-alert" time such that 2.5s escalate to 2s after that time has passed

** Sev-3
- Standard priority for most requests
- Regular workflow processing
- Daily digest notifications

** Sev-4
- Nice-to-have, no specific timeline
- "Someday/maybe" category
- Weekly summary only

* Notification System

** Real-time Notifications
- New ticket submissions
- Status changes on your tickets
- Comments on tickets you're involved with
- Approaching due dates


** Communication Channels
- In-app notifications
- Email summaries
- SMS for urgent items (optional)
- Push notifications for mobile apps


* Mobile Considerations

** Offline Capability
- Cache recent tickets for offline viewing
- Queue actions when connectivity returns
- Sync conflicts resolution

* Security & Privacy

** Access Control
- Personal tickets remain private
- Friend permissions are granular
- Guest submissions have limited visibility

** Data Protection
- Ticket data encryption
- Secure friend invitation system
- Optional ticket expiration/archiving

* Future Enhancements

** Integration Possibilities
- Calendar sync for due dates
- Email ticket creation
- Slack/Discord bot interface
- GitHub integration for code-related tasks

** Advanced Features
- Ticket templates for common requests
- Automated ticket routing based on keywords
